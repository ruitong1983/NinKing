# Part 0: 核心定义

## 一句话

> **手牌 9 张，排列成 3 组 × 3 张（影≤瞬≤滅升序约束），AI 辅助排列，三组同时计分。搭载忍者牌、附魔卡、星图卡、喜、秘仪五大系统，闯过 8 Antes × 3 Blinds 通关。**

## 核心参数速查

| 参数 | 值 |
|------|-----|
| 手牌 | 9 张 |
| 每组 | 3 张（影/瞬/滅，共 3 组） |
| 排列约束 | 牌力(影) ≤ 牌力(瞬) ≤ 牌力(滅) — **硬约束** |
| 出牌次数 | **3** / Blind |
| 换牌次数 | **2** / Blind，每次换 **1~3 张** |
| 交换位置 | 不限次（玩家通过交换覆盖 AI 排列） |
| 忍者牌槽位 | 5（可通过 Negative 版本扩展） |
| 初始金币 | **$8** (v3.1: $4→$8) |
| 利息 | 每 $5 余额 +$1，上限 $5 |
| 牌组 | 5 种（标准/暗夜/赤阳/混沌/龙脉） — Phase D |
| 存档 | 纯永久死亡，仅保留解锁进度 |

# Part 1: 核心循环

## 1.1 Blind 内循环

```
发 9 张手牌
    │
    ├── AI 自动排列 → 展示 3 组 + 预期忍気 + 喜触发提示
    │       │
    │       ├── [交换] 拖动牌在组间/组内移动（不限次）
    │       │       覆盖 AI 排列，原地重评估（不自动触发 AI 重排）
    │       │       实时约束验证，列牌型同步更新
    │       │
    │       ├── [换牌] → 选 1~3 张丢弃 → 重抽 → AI 重排
    │       │       每 Blind 2 次
    │       │
    │       ├── [AI 重排] 点击按钮 → AI 重新优化横向排列
    │       │
    │       └── [出牌] → 计分 → 抽 9 张 → 下一轮
    │
    └── 忍気达标 → 过关（进入商店）
        出牌次数归零且未达标 → 失败（永久死亡）
```

### v3.1 交互决策

| 操作 | 消耗 | 解决的问题 |
|------|------|-----------|
| 交换 | 不限次 | 微调排列结构，覆盖 AI 建议，同时影响列牌型 |
| AI 重排 | 不限次 | 恢复 AI 最优横向排列（不覆盖列优化决策） |
| 换牌 | 2 次/Blind, 1~3 张/次 | 改善手牌质量 |
| 出牌 | 3 次/Blind | 确认计分 → 下一轮 |

> **设计决策 (v3.2):** 交换后不再自动触发 AI 重排——玩家手动调整位置后原地保留，AI 重排改为独立按钮供玩家主动调用。交换可同时影响横向约束和纵向列牌型，列优化是玩家技能表达的核心路径。

# Part 2: AI 自动排列

## 2.1 算法

- 暴力枚举 1680 种排列（C(9,3)×C(6,3)）
- 过滤合法排列（满足影≤瞬≤滅约束）
- 在合法排列中选分数最高的
- 性能: < 1ms（GDScript 单帧）

## 2.2 牌力编码

强度值（`strength`）用于约束校验：`头(影) ≤ 中(瞬) ≤ 尾(滅)`。

```
strength = hand_type_idx × 1000 + high_rank × 10 + mid_rank + suit_tiebreak

hand_type_idx: 豹子=5, 同花顺=4, 同花=3, 顺子=2, 对子=1, 散牌=0
suit_tiebreak: ♠=4, ♥=3, ♦=2, ♣=1
```

`hand_type_idx × 1000` 确保牌型优先于点数（最大点数贡献 = 154 < 1000）。
约束校验仅用 `strength` 值比较，不依赖具体牌型枚举。

## 2.3 规则型忍者牌的 AI 适配

持有规则变更型忍者牌时，AI 对应调整过滤条件：

| 忍者牌 | 效果 | AI 行为 |
|--------|------|---------|
| n_r02 均衡之印 | 三行非散排时所有行列组+50筹码 | 无限制（仅计分加成，不约束AI） |
| n_r03 独尊之印 | 前两行不计分，只对第三行计分×3 | 无限制（仅尾行×3，不约束AI） |

> n_r01 自由之印已删除——解除约束移除核心 Puzzle，无趣。

## 2.4 AI 排列的玩家覆盖

- AI 给出最优排列建议（仅优化横向 3 组）
- 玩家可通过**交换**（不限次）手动调整 → 原地重评估，不触发 AI 重排
- **AI 重排按钮**可随时恢复 AI 最优横向排列
- 约束校验器实时反馈：绿箭头 = 满足，红箭头 = 违规
- 出牌按钮仅约束满足时可用（默认为升序，Boss「反目」改为降序）
- 列牌型仅在玩家交换后变化，AI 不优化列

# Part 3: 计分系统 (v5.0)

## 3.1 公式（行+列双维度加总）

```
行分 = (card_chips + hand_chips + ench_chips + ninja_chips)
     × (hand_mult + ench_mult + ninja_mult)
     × ∏(ninja_x_mult) × ∏(card_x_mult) × ∏(group_xi)

列分 = 同公式，牌来自纵向列（col_j = [影[j], 瞬[j], 滅[j]]），散牌列跳过(贡献0)

总分 = 影 + 瞬 + 滅 + 列0 + 列1 + 列2
最终 = 总分 × ∏(global_xi)
```

### 关键规则

| 规则 | 说明 |
|------|------|
| **行+列双维度加总** | 三行和三列（非散牌）各自独立计分，然后加总。行和列共享同一批卡片的牌面/附魔/封印属性（每张卡在行和列各贡献一份） |
| **散牌列跳过** | 列牌型为散牌时整列贡献 0（不计牌面筹码也不计牌型） |
| **忍者/附魔跟组走** | 行和列各自独立收取忍者效果。无条件忍者三行+三列各得一份(≤6份)。条件型忍者的 hand_type 条件同时匹配行和列，group 条件只对行生效 |
| **星图升级** | 行和列共享同一套星图等级（升级"豹子"→行和列的豹子都变强） |
| **列基础值** | 和行使用相同的牌型基础值(chips+mult)，不做额外缩放 |
| **全局喜** | 最终 ×mult 在行+列加总后乘上去（和 v4.0 一致） |
| **组级喜** | 三清/三顺清 → 行和列各 ×2/×3；顺清打头 → 仅影行 ×2 |

### 3.1a 列牌型 (v5.0 重写)

列不再提供 ×mult 累乘，改为独立的 chips×mult 加总项，和行平级。

| 列牌型 | chips | mult |
|--------|-------|------|
| 豹子 | 100 | 8 |
| 同花顺 | 50 | 5 |
| 同花 | 30 | 4 |
| 顺子 | 20 | 3 |
| 对子 | 10 | 2 |
| 散牌 | 跳过（不计分） |

- 列值吃星图升级（和行共享同一 HandType3 升级表）
- AI 只优化横向排列，列留给玩家手动交换优化
- 列改加法后优化列的边际收益降低，但仍是有意义的技能表达

# Part 4: 喜系统

## 4.1 机制

- 每次出牌后自动检测 9 张牌的全局模式
- 每个喜独立检测，触发即叠加（扁平模型）
- 无分类互斥、无上限、无优先级纠结
- 合系列（三合/双合/一合）互斥，只取最高档
- Debug 场景下喜即时检测：发牌/随机/AI 重排后左边栏 ColXiLabel 即刻更新

>

>
> 双壁与合系列可能同时触发（左列右列同型意味着合系列至少+1），但合系列≥2时双壁自然成立概率反而不高。
>

>
> 连环是"棋盘根喜"——一旦成立，12 条正交邻边全异色自动带出四角+倒影+对影（×3 ×3 ×2 = ×18 套餐）。与全黑/全红/中十字天然互斥。构造难度极高，属于玩家手动交换才能达成的高技能表达。
>
> 廿一点取自 21 点情怀——7+7+7 同时触发豹子+廿一点，是设计上的彩蛋峰值。
>
> 天九（天地人三才）取牌九至尊，与三等（水平相等）形成平衡型的双分支——三等求等量，天九求互补，两者天然互斥。
>
> 压牌取锄大地精髓——各行 rank 区间无交叠，与昇龍（牌型递增）正交：昇龍看型，压牌看值，可同时触发。
>
> 文武取牌九文武分类——文牌对称工整，武牌杂乱粗犷。豹子=牌型天花板(文)，散牌=牌型地板(武)，天壤之别同炉共冶。与背水可同时触发（武行=尾墩散牌→文武×背水=×4×4套餐）。
>
> 一气取牛牛"有牛"——3张凑10的倍数即"有牛"。貧打(2-3-5)天然chip=10，貧打一气双响×4×3=×12。K-K-K同时触发豹子+均爵+一气=×2×3×3=×18彩蛋峰值。与廿一点互斥（21不是10的倍数）。
>
> 无将取桥牌最高叫品——放弃将牌特权，四色平等，纯靠牌力。唯一可行分布 3-2-2-2（每色≥2）。与全同花/全黑/全红/三清互斥（集中 vs 分散），与连环/满堂/均爵可共存。是当前唯一检测"花色分散"的喜。
>
> 长套取桥牌长套美学——一门六张独大，三门各一陪桥（6-1-1-1）。与全同花/无将/空门/三清均互斥（各美其美），与非颜色类喜全可共存。无将(均衡)×4 ↔ 长套(集中)×5 形成花色分布谱系双极。
>
> 无忧角取围棋守角定式——3×3 九宫格分 4 个 2×2 子方格，任一全同色即触发。是第一条"子方格"维度检测，填补了行列对角之外的微观结构空白。
>
> 慢打取德扑经典战术——前段散牌假装弱牌（影=散牌），后段亮出怪兽牌收网（滅=豹子）。影弱滅强，制造"以为我要输→突然翻盘"的心理波动。与背水互斥（背水要求尾墩散牌），与文武可同时触发（文=豹子，武=散牌，慢打的两端正好构成文武）。
>
> 四对半取十三张"六对半"压缩至 9 张版——4 对完好 + 1 张孤牌（2-2-2-2-1）。4 对分配进 3 行 → 至少 1 对被拆散跨行，结构上必然出现 2 行对子 + 1 行可能散牌。无豹子/全三条/四张等 rank 集中型喜，与均爵/至尊/颜色系可共存。

# Part 5: 消除模式 (Clean Mode) — Balatro 风消除

> **设计来源:** Grill 10 轮决策, 2026-06-23 | **规格书:** [`specs/clean-mode-design.md`](../09-mgmt/specs/clean-mode-design.md)
> **实施状态:** Phase 1 核心玩法 ✅ | Phase 2 计分配置 ✅ | Phase 3 UI+VFX ⬜

## 5.1 核心概念

消除模式是 NinKing 的第二种游戏模式。与比鸡模式共享结界/封印/金币/商店/忍者主体框架，但核心玩法完全不同：

| 维度 | 比鸡模式 (bi_ji) | 消除模式 (clean) |
|------|------------------|-----------------|
| 核心操作 | 排列 9 张牌为 3 组（头/中/尾） | 相邻交换，三连消除 |
| 操作限制 | `plays_per_seal=3`（出牌次数） | `clean_swaps_per_seal=5`（交换次数） |
| 计分单位 | 组（头/中/尾）+ 列 + 喜 | 消除波次（chain wave） |
| 约束 | 升序约束（头≤中≤尾） | 相邻约束（只能交换相邻卡） |
| 补充机制 | 无（9 张固定排列） | 重力补牌 + 连锁消除 |
| 喜系统 | ✅ 全局喜 / 组喜 / 列喜 | ❌ 无喜系统 |
| 牌型手役 | ✅ 同花/顺子/豹子等 | ✅ **豹子/同花顺/同花/顺子**（行/列消除检测） |

## 5.2 核心循环

```
封印开始
  ↓
初始发牌: peek 9 张 → 验证无四类三连（豹子/同花顺/同花/顺子）→ 有效则落牌
  ↓
PLAYING 状态
  ↓
玩家操作: 点击相邻两张牌 → 交换
  ↓
检测: 是否有三连 (豹子/同花顺/同花/顺子)?
  ├── 无三连 → 交换生效 → swaps_remaining -1 → 等待下一次交换
  └── 有三连 → 进入连锁消除流程
         ↓
      连锁消除（多波次）: 匹配→移除→重力补牌→再检测
         ↓
      swaps_remaining -1
         ↓
      判定:
        ├── current_score ≥ target_score → 封印突破
        ├── swaps_remaining ≤ 0 → GAME_OVER
        └── 继续 → PLAYING 等待下一次交换
```

## 5.3 消除计分公式

### 5.3.1 基础波次计分

```
每波原始分 = Σ(匹配组 rank_value × 10) × chain_multiplier
chain_multiplier = 1 + (chain_level - 1) × 0.5   ← chain_level 从 1 开始
```

- rank_value: 2=2, 3=3, ..., 10=10, J=11, Q=12, K=13, A=14
- 十字消除（行列交叉）奖励: +50 额外分（算入当波原始分）
- 硬编码连锁上限: **10 波**（防御死循环）

### 5.3.2 含忍者计分（完整公式）

由 `ScoreCalculator.calculate_clean()` 实现：

```
输入: hand(9张牌), chain_waves(各波次数据), ninjas(持有忍者), gold(当前金币)

Step 1 — 忍者效果过滤 (_is_ninja_valid_for_clean):
  - 无条件忍者   → ✅ 生效
  - hand_type/group/col_hand_type/xi 条件 → ⛔ 跳过（消除模式无对应概念）
  - 经济效果 (mult_per_gold, x_per_gold) → ✅ 生效

Step 2 — 忍者效果累加:
  累加 each valid ninja:
    add_chips_total += eff.add_chips
    base_multiplier += eff.add_mult
    x_stack.append(eff.x_mult)          // beyond ×1
    base_multiplier += floor(gold/5) * mult_per_gold  // 金剛力
    x_stack += 每 $15 增 ×2, 最多 3 次  // 黄金律
    gold_on_swap += eff.gold_on_play    // 每次 swap 产金
    extra_swaps += eff.extra_plays      // 额外交换次数
    if only_one_play: only_one_swap = true

Step 3 — 各波次计分:
  for each wave:
    raw = Σ(match.rank × 10) × chain_multiplier
    wave_score = raw + add_chips_total    // 忍 chips 加到此波
    swap_score += wave_score

Step 4 — 乘算:
  swap_score = ceil(swap_score × base_multiplier)
  for each x in x_stack:
    swap_score = ceil(swap_score × x)

输出: {swap_score, wave_scores, gold_on_swap, extra_swaps, only_one_swap}
```

### 5.3.3 忍者效果映射

| 效果字段 | 比鸡语义 | 消除语义 |
|---------|---------|---------|
| `add_chips` | 加到组 chips 上 | 直接加到每波波次分数上（每波都加） |
| `add_mult` | 加到组 mult 上 | 加法到 base_multiplier（该次 swap 末尾乘算） |
| `x_mult` | 乘到组 x_stack | 乘法乘到该次 swap 总分上 |
| `extra_plays` | +1 出牌次数 | +1 swaps_remaining |
| `gold_on_play` | 出牌产金 | 每次 swap 首波产金 |
| `only_one_play` | 强制仅 1 次出牌 | 强制仅 1 次交换 |
| `mult_per_gold` | 每 $5 +1 mult | ✅ 同左（金剛力） |
| `x_per_gold` | 每 $15 ×2 | ✅ 同左（黄金律） |

### 5.3.4 条件型忍者处理

**消除模式中，仅无条件忍者 + 经济型忍者生效。** 约 60% 带 `hand_type`/`group`/`xi` 条件的忍者被静默跳过。商店中仍然可见，但不会触发。未来考虑引入消除模式专属忍者池。

## 5.4 技术架构

### 5.4.1 关键文件

| 文件 | 职责 |
|------|------|
| `scripts/ninking/clean_controller.gd` | 核心引擎: 网格操作、匹配检测、连锁执行、交换验证、计分（基础分） |
| `scripts/ninking/clean_layout_generator.gd` | 初始布局: peek 9 张 → 验证四类三连 → 无效则 shuffle 重试（无上限，不消耗牌） |
| `scripts/ninking/score/score_calculator.gd` | `calculate_clean()` — 含忍者效果的消除计分管线 |
| `scripts/ninking/game_state.gd` | 模式分支: `game_mode` 字段、`_cascading` 连锁锁、`swaps_remaining` 管理 |
| `scripts/ninking/ui/game_manager.gd` | 连锁动画循环 `_resolve_clean_chain()` |

### 5.4.2 Cascading 锁

连锁消除期间（`gs._cascading = true`），所有输入被锁定：
- `swap_cards()` 检查 `_cascading`，true 时直接 return
- 连锁结束后 `_cascading` 置回 false
- 通过 `prepare_chain_wave()` → `apply_chain_wave()` 循环实现波次间状态更新

### 5.4.3 `CleanController` vs `SealController`

**独立不继承。** 消除模式的 prepare/finalize 流程完全不同（无分组、有连锁）。共用部分（`_complete_seal`、利息计算）在 `clean_controller.gd` 中独立实现。
